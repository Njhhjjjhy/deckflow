## Problems encountered and how they were solved

### 1. macOS `sed` multi-line substitution fails
- **Symptom:** `sed -i '' '/\.text-chart-page__heading/{N;s/font-weight: 700/font-weight: 600/}'` returned "bad flag in substitute command: '}'"
- **Root cause:** macOS `sed` handles multi-line patterns differently than GNU sed; the `{N;...}` construct requires different syntax
- **Fix:** Used the `Edit` tool directly to make targeted edits to each specific file

### 2. Remaining `#CCCCCC` color violations after bulk sed
- **Symptom:** Final verification grep found 3 remaining `color: #CCCCCC` in map-text CSS files
- **Root cause:** The sed pattern `^\s*color.*#CCCCCC` didn't match because the actual lines had slightly different surrounding context
- **Fix:** Direct targeted `sed -i '' 's/color: #CCCCCC/color: #ADADAD/g'` on the three specific files

### 3. Nested `<tr>` in BlocksLibrary
- **Symptom:** `BlockRow` returned a `<tr>`, but the parent `.map()` also wrapped it in a `<tr>`, creating invalid HTML
- **Root cause:** Oversight when designing the component split between `BlockRow` and the parent table
- **Fix:** Removed outer `<tr>` wrapper from `.map()`, added `isFirst: boolean` prop to `BlockRow` to handle the `borderTop` style that was previously on the outer wrapper

### 4. Duplicate `</div>` in PresentationEditor
- **Symptom:** After inserting the outer `selectedPage?.reusableBlockId ?` ternary, verified the file had two consecutive `</div>` closing tags at lines 176–177
- **Root cause:** The `Edit` replacement of the closing ternary accidentally added `</div>` at the end of the `new_string`, but the original file already had `</div>` after the `)}` at that location
- **Fix:** Second `Edit` to remove the duplicate `</div>`

### 5. TypeScript strict mode / circular import concern
- **Symptom/Concern:** Both stores needed to call each other's actions (presentationStore.syncBlockContent called from blocksStore.updateBlock, and vice versa)
- **Root cause:** Zustand stores are initialized at module load time; circular ES module imports can cause initialization-order issues
- **Fix:** Adopted component-level coordination pattern — components call both store actions in sequence rather than having stores import each other. This is architecturally cleaner and avoids the circular dep entirely.

## Architecture and design notes

### Cross-store communication pattern
The presentation store and blocks store do NOT import each other. All cross-store coordination happens at the component level:
- `PageList`: calls `addPageFromBlock(block)` (pres store) + `addUsage(block.id, presId)` (blocks store)
- `BlocksLibrary.handleSave`: calls `updateBlock(...)` (blocks store) then `syncBlockContent(...)` (pres store)
- `BlocksLibrary.handleDeleteConfirm`: calls `unlinkBlock(...)` (pres store) + `removeUsage(...)` (blocks store) + `deleteBlock(...)` (blocks store)
This avoids circular module dependencies and keeps stores single-responsibility.

### Block content display in editor
When a page has `reusableBlockId`, the `PresentationEditor` shows `BlockBanner` instead of the page's editor component. The page's `content` field still holds a copy of the block's content (kept in sync by `syncBlockContent`), so the `SlidePreview` renders correctly without any special handling.

### Route structure
```
#/           → PresentationEditor
#/blocks     → BlocksLibrary (list view)
#/blocks/ID  → BlocksLibrary (edit form open for block ID)
```
The `BlocksLibrary` component accepts `editBlockId?: string` and opens the edit form via `useEffect` when this prop is set.

### Block content editor (BlockForm.tsx)
Content fields are rendered dynamically by iterating over `Object.entries(content)`. Field type is determined by:
1. `isTranslatableField(value)` (has `en` + `translationStatus`) → LanguageTabs
2. `isImageField(key)` (key contains image/icon/photo) → read-only note
3. `isColorField(key)` or `isSkippedField(key)` → hidden entirely
4. `isJsonDataField(key)` (ends in "Data") → monospace textarea
5. Otherwise → plain text input

### CSS naming convention in templates
All template CSS uses BEM: `.{template-name}__element`. E.g., `.data-table-page__heading`, `.cover-page__hero`. Each template's CSS is fully self-contained (no shared classes) — intentional for isolation, at the cost of repeated header bar rules.

### The map-text three-file split
`MapTextCardPage.tsx` (Mode A: map + org cards), `MapTextListPage.tsx` (Mode B: map + property list + summary table), `MapTextOverlayPage.tsx` (Mode C: map + floating callouts). All three are dispatched from `SlidePreview.tsx` for pages where `page.type === 'map-text'`, selected by `page.content.mapTextMode` ('A', 'B', or 'C'). This is intentional per CLAUDE.md spec — not a cleanup candidate.
